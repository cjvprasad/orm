01. Primary Relationship Mappings

1. Uni-directional One to One

The simplest relationship. Only the source entity contains a foreign key to the target.

@Entity
public class User {
  @Id @GeneratedValue private Long id;
  @OneToOne(cascade = CascadeType.ALL)
  @JoinColumn(name = "profile_id", referencedColumnName = "id")
  private Profile profile;
}
2. Bi-Directional One to One

Both entities reference each other. One side MUST use mappedBy to identify the non-owning side.

// Owner (Physical FK in DB)
@OneToOne
@JoinColumn(name = "person_id")
private Person person;
// Inverse (Java-only link)
@OneToOne(mappedBy = "person")
private Passport passport;
3. Uni-Directional One to Many

A single parent has a collection of children. Always use @JoinColumn to avoid an automatic join table.

@OneToMany(cascade = CascadeType.ALL)
@JoinColumn(name = "order_id") // FK is stored in the Child table
private List<Item> items;
4. Bi-directional One to Many (Foreign Key Mapping)

The standard approach. The 'Many' side owns the relationship because it holds the dept_id FK.

// In Employee.java (The Owner)
@ManyToOne
@JoinColumn(name = "dept_id")
private Department department;
5. Bi-Directional Many to Many (Hibernate Managed Join Table)

Hibernate hides the link table from the Java code. Use this for simple associations.

@ManyToMany
@JoinTable(
  name = "student_course",
  joinColumns = @JoinColumn(name = "student_id"),
  inverseJoinColumns = @JoinColumn(name = "course_id")
)
private Set<Course> courses;
6. Bi-Directional Many to Many (User Managed Join Table Object)

Required when the "Relationship" itself has data (e.g., Score, Timestamp, or Quantity).

// The middle entity (Link Table Object)
@Entity
public class Enrollment {
  @ManyToOne private Student student;
  @ManyToOne private Course course;
  private int grade; // User-managed extra data
}

02. Ownership & Performance Logic

Ownership Strategy

Ownership determines which table the FK is stored on. Typically, the Many-side owns the link.

Example: In a User-Role setup, manipulate the Roles through the list of Roles on a User by using Cascade. This is O(1) in terms of DAO logic compared to iterating through roles.

Fetch Strategy (LAZY vs EAGER)

Lazily fetched collections are safer. EAGER loading can lead to massive performance penalties and Heap memory issues.

Manual Load: If you need a collection just for one call, use:
Hibernate.initialize(person.getRoles());
Advanced Case Study: Heap Crash & Batching

For bulk jobs (e.g., <80,000+ inserts), standard Hibernate caching will crash the system. Always flush and clear:

for (int i = 0; i < objects.size(); i++) {
  em.persist(objects.get(i));
  if (i % 50 == 0) {
    em.flush(); em.clear(); // Fixes memory leak
  }
}
