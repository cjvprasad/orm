1️ Determining the Owning Side (Bi-Directional Relationships)
TopicExplanation
What is Owning SideSide that contains the foreign key in the database
Why It MattersDetermines which table stores the FK and controls updates
Hibernate BehaviorHibernate works either way; only FK location changes
General RuleThe side with the collection usually owns the relationship
Practical GuidelineOwn the relationship from the object you manipulate most
ExampleUser has List<Role> → Role owns the relationship
Cascade LocationUsually defined on the owning side
2️ Fetch Type Decision (LAZY vs EAGER)
TopicWhen to Use
fetchType.LAZYWhen collection is not always needed
Performance ImpactVery high performance benefit
Default BehaviorHibernate defaults to LAZY for collections
fetchType.EAGERWhen related data is almost always required
Performance CostHigh – extra checks and joins
WarningDo NOT use EAGER just to fix LazyInitializationException
Design SmellUsing EAGER to fix lazy issues indicates poor design
3️ Handling LazyInitializationException (Correct Way)
ApproachDescription
Bad PracticeChanging fetchType to EAGER
RecommendedInitialize collection explicitly in Service layer
Example MethodHibernate.initialize(entity.getCollection())
Transaction ScopeMust be inside @Transactional
DownsideNo control over fetch strategy

TopicExplanation
PurposeControl how Hibernate fetches collections

5️ Determining Cascade Direction
TopicExplanation
What is CascadeAutomatically propagates persistence operations
Common MistakeCascading in both directions unnecessarily
GuidelineCascade from parent to child
ExampleUser → Roles cascade
ReasonParent controls lifecycle
Typical Settingcascade = CascadeType.ALL on owning side
6️ Cascade Efficiency Comparison
ApproachCost / Behavior
Save User with Roles1 User insert + N Role inserts
Save Roles IndividuallyN Role inserts + N User updates
Time ComplexityO(1) vs O(N)
RecommendationAlways save parent with children
7 Orphan Removal (orphanRemoval = true)
TopicExplanation
What It DoesDeletes child if removed from parent collection
When to UseTrue ownership relationships
When NOT to UseShared reference data (e.g., Roles)
ExampleRemoving Role from User should NOT delete Role
RiskAccidental data loss

@OneToOne (Bi) delete Foo (no cascade) UPDATE bar SET foo_id=NULL
DELETE FROM foo WHERE id=? 		Relationship broken, Foo deleted
@OneToOne (Bi) delete Foo (cascade REMOVE) DELETE FROM foo WHERE id=?
DELETE FROM bar WHERE id=? 		Foo and Bar deleted
@OneToMany (Uni, Join Table) delete Foo DELETE FROM FOO_BAR WHERE foo_id=?
DELETE FROM foo WHERE id=? 		Join table rows removed, Bars remain
@OneToMany (Bi, FK in Bar) delete Foo (no cascade) UPDATE bar SET foo_id=NULL
DELETE FROM foo WHERE id=? 		FK cleared on children, Foo deleted
@OneToMany (Bi, cascade REMOVE) delete Foo DELETE FROM bar WHERE foo_id=?
DELETE FROM foo WHERE id=? 		Children and Foo deleted
@ManyToMany (Bi, Join Table) delete Foo DELETE FROM FOO_BAR WHERE foo_id=?
DELETE FROM foo WHERE id=? 		Only links removed, Bars remain
@ManyToMany (Bi, Join Table) delete Bar DELETE FROM FOO_BAR WHERE bar_id=?
DELETE FROM bar WHERE id=? 		Only links removed, Foos remain
